Conversation Summary: Face OS Verification & Renderer Fixes

1. Initial Context & Investigation

  • Initial Request: The user requested an investigation into why identity verification was failing on the processed output video from the face_os pipeline, specifically noting that the output was "total shit" and failing to verify. The constraint was to diagnose the issue without making code changes initially.
  • Diagnosis Run:
    • Integration tests passed.
    • A test run on 150 frames of input/video.mp4 showed that the face matcher successfully verified the original video (899/900 frames) and the first 3 frames of the output (which used the alpha fallback path).
    • From frame 4 onwards, the physical renderer path was engaged, and the face matcher reported No face detected.
  • Root Cause Identification: The frames processed by the physical renderer were rendering almost completely black (mean pixel value ~1.0/255 vs original ~60.5/255). The HOG face detector failed due to a lack of gradients.
  • Three Mathematical Bugs Identified:
    1. Double-Attenuation of Illumination: In pipeline.py, the lighting components (ambient and diffuse) were pre-scaled by the shading mean. In physical_renderer.py, these components were multiplied again by irradiance (which is also the shading mean), effectively squaring the fractional value (e.g., 0.25 × 0.25 = 0.0625) and causing a massive brightness drop.
    2. Shading Magnitude Discrepancy: The physical renderer computed shading using fractional weights, resulting in a significantly lower shading mean (~0.03) without scaling it back to the original shading energy.
    3. Flawed Energy Conservation: The _apply_energy_conservation function compared the rendered frame's energy against the raw albedo mean (~0.74) instead of the expected output energy (albedo × shading, ~0.26). Thus, it never recognized the frame as being "too dark" and failed to scale it up.

2. Applying Fixes

  • User Instruction: Apply the fixes, tighten/rewrite integration tests, and use parallel agents.
  • Execution Strategy: Spawns were attempted for parallel agents to fix the codebase and rewrite the tests. However, the agents hit API quota limits (RESOURCE_EXHAUSTED).
  • Manual Intervention: I applied the fixes manually to the codebase:
    • Fixing physical_renderer.py:
      • Removed the redundant irradiance multiplication from _compute_ambient, _compute_diffuse, and _compute_specular.
      • Updated _apply_energy_conservation to accept a target_energy parameter (calculated as albedo × shading) and added logic to scale the frame up if the output energy ratio drops below 0.5.
    • Rewriting Tests: Deleted the old test_integration.py and wrote a new suite focusing heavily on the physical renderer's outputs. Added strict assertions requiring physical frames to not be black (mean > 20/255), to have a valid brightness ratio compared to alpha frames, and to successfully pass a face detector.

3. Testing & Current Status

  • Test Results:
    • The synthetic unit tests for energy conservation passed successfully.
    • The full integration tests on the real video failed.
  • Ongoing Issue Analysis: While the energy conservation fix improved the output brightness slightly (mean pixel value went from 1.0/255 to 8.1/255), it still failed the >20.0/255 threshold.
  • Next Steps Identified (Before Interruption): The pipeline-side lighting estimation in pipeline.py (around lines 1955-1959) is pre-scaling ambient by 0.3 × shading_mean and diffuse_intensity by 0.8 × shading_mean. This leaves the base render output significantly below the target energy before conservation is applied. The pipeline's lighting model values need recalibration to ensure the renderer receives sufficient baseline energy.
